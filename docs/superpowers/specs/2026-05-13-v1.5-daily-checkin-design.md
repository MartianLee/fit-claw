# v1.5 Sub-project 1: Daily Checkin Design

**Status:** approved
**Date:** 2026-05-13
**Parent plan:** docs/superpowers/plans/2026-05-03-fit-claw-v1.md (§6.4)
**Scope:** First of three v1.5 sub-projects. Subsequent: Personalized Prediction (§6.5), Threshold Alarms (§6.6). Those are out of scope here.

## Goal

Let an agent log and query daily sleep/condition/notes through `/tools/*`, so 4–8 weeks of dog-fooding data accumulates before the prediction and alarm sub-projects are built on top of it.

## Why now

- Schema (`daily_checkins`) already exists from v1's initial migration.
- Prediction sub-project needs historical condition/sleep data to be useful; the longer this is delayed, the more it pushes out v1.5 value.
- No dashboard work needed yet — visualization decisions defer until prediction + alarm shape the actual use case.

## Non-goals

- Dashboard cards or charts for checkins (defer until after prediction sub-project).
- Reminder cron / agent webhook prompting the user to check in (belongs in alarms sub-project).
- Importing sleep from Apple Health / HealthKit (v2).
- Multi-user support (single-user self-hosted assumption holds).

## Tools

Two endpoints, both bearer-auth via the existing middleware in `src/auth/bearer.ts`. Same Zod-validator (`zv`) wrapper for unified error shape (`{error:{code:"invalid_input",message}}`).

### `POST /tools/log_checkin`

**Input:**

```ts
{
  date?: string | null      // YYYY-MM-DD; default = "today" in user's TZ
  sleep_hours?: number | null      // 0–24, patch
  condition_score?: number | null  // 1–5 integer, patch
  notes?: string | null            // append; null = clear
}
```

**Semantics:**

- Resolves `user_id` from bearer auth context.
- Resolves `date` from `users.timezone` if not provided (`Intl.DateTimeFormat('en-CA', {timeZone: tz}).format(new Date())`).
- Validates `today - 90 ≤ date ≤ today` in user TZ. Out of range → 400.
- Requires at least one of `sleep_hours`, `condition_score`, `notes` to be present (any value including `null`). Empty body → 400 `"at least one of sleep_hours, condition_score, notes required"`.
- Upserts on `(user_id, date)`:
  - `sleep_hours`, `condition_score`: patch. Undefined = leave alone. `null` = clear column to NULL. Number = set.
  - `notes`: append. `null` = clear column to NULL. String = append.
    - Append format: `[HH:MM] <text>` with `HH:MM` formatted in `users.timezone`.
    - Dedupe: if new text equals the existing trailing line's text (after stripping the `[HH:MM] ` prefix), skip the append. This makes retries idempotent.
    - Length cap: total `notes` column must stay ≤ 4000 chars after append. Over cap → 400 `"notes exceed 4000 chars; clear before continuing"`.
- Returns the resulting full row.

**Output:**

```ts
{
  id: number
  user_id: number
  date: string
  sleep_hours: number | null
  condition_score: number | null
  notes: string | null
}
```

### `POST /tools/query_checkins`

**Input:**

```ts
{
  date_from?: string  // YYYY-MM-DD inclusive, default = today - 30
  date_to?: string    // YYYY-MM-DD inclusive, default = today
}
```

- `date_from` / `date_to` use the same `YYYY-MM-DD` regex validation.
- Range cap: `date_to - date_from ≤ 365 days`. Over → 400.
- Returns rows in ascending date order.

**Output:**

```ts
Array<{
  id: number
  user_id: number
  date: string
  sleep_hours: number | null
  condition_score: number | null
  notes: string | null
}>
```

## Validation rules (Zod)

```ts
const DATE_RE = /^\d{4}-\d{2}-\d{2}$/

const LogCheckinSchema = z
  .object({
    date: z.string().regex(DATE_RE).nullish(),
    sleep_hours: z.number().min(0).max(24).nullish(),
    condition_score: z.number().int().min(1).max(5).nullish(),
    notes: z.string().max(2000).nullish(),
  })
  .refine(
    (v) =>
      'sleep_hours' in v || 'condition_score' in v || 'notes' in v,
    { message: 'at least one of sleep_hours, condition_score, notes required' },
  )
```

Note: `notes` per-call is capped at 2000 chars; the stored column cap of 4000 is enforced after the append computation.

```

  lib/

    checkins.http.test.ts  # auth + happy path + error shape
  lib/
    time.test.ts        # tz helper edge cases
```

No new migration. `daily_checkins` and `users.timezone` already exist.

## Boundaries

- `src/domain/checkins.ts` is the only file that touches the `daily_checkins` table.
- `src/lib/time.ts` is the only place that constructs date strings from `Date` + tz.
- `src/tools/checkins.ts` does no business logic; it validates input, resolves `user_id` from context, calls domain, returns JSON. Mirrors `src/tools/workouts.ts` shape so the agent-facing surface stays consistent.

## Error handling

- All input validation errors return `{error:{code:"invalid_input",message}}` 400 via the `zv()` wrapper or domain-thrown `HttpError`. Defines a small `HttpError(status, code, message)` already used for similar cases (verify in `src/lib/http.ts`; reuse, don't add a new one).
- Auth errors continue through bearer middleware (401).
- No 5xx paths beyond unexpected DB errors (let Hono's onError handle).

## Testing

TDD per task in the implementation plan. Key cases the test suite must cover:

1. `logCheckin` creates row when none exists for that date.
2. Second call with `sleep_hours` only patches sleep, preserves prior `condition_score`.
3. `condition_score: null` clears that column.
4. `notes` appends with `[HH:MM]` prefix in user tz.
5. Retried call with identical notes does not duplicate the line.
6. `notes: null` clears the column.
7. Length cap rejects when append would exceed 4000.
8. Empty body rejected.
9. Date outside window rejected (today + 1, today - 91).
10. Invalid scalar bounds rejected (`sleep_hours: 25`, `condition_score: 0`).
11. `queryCheckins` returns ascending range with defaults applied.
12. HTTP smoke test: 401 without bearer; happy-path 200; bad input returns unified error shape.
