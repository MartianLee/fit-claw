# Fit Claw v1.5 Prediction and Alarms Design

## Goal

Implement personalized exercise performance prediction and threshold alarms before daily checkin work.

This slice gives agents two reliable capabilities:

- Ask for exercise-specific future performance estimates.
- Register threshold rules that are evaluated daily and delivered through agent webhook with Telegram fallback.

## Non-Goals

- No dashboard UI changes in this slice.
- No daily checkin correlation yet.
- No advanced ML model or external model dependency.
- No always-on scheduler requirement inside the HTTP server. The evaluator is exposed as reusable domain code and a one-shot job that self-hosters can schedule daily.

## Prediction Model

### Inputs

Prediction uses existing workout data:

- `workout_sessions.started_at`
- `workout_entries.exercise_id`
- `workout_sets.weight_kg`
- `workout_sets.reps`

The estimated one rep max uses the existing Epley calculation:

```ts
estimated_1rm = weight_kg * (1 + reps / 30)
```

For each date, only the highest estimated 1RM is used.

### Trend

For each `(user_id, exercise_id)`, fit a simple linear regression over the last 12 weeks of daily estimated 1RM points:

- `x`: weeks since first point in the fit window
- `y`: daily max estimated 1RM
- output: `slope_per_week`

If fewer than two points exist, `slope_per_week` is `0`.

### Rest Decay

The model estimates personal rest decay from historical gaps:

- Find consecutive training dates for the same exercise.
- A gap is eligible when `gap_days >= 7`.
- If the next estimated 1RM is below the previous estimated 1RM, calculate a tau candidate:

```ts
tau_days = gap_days / -ln(next_est_1rm / previous_est_1rm)
```

- Use the average valid candidate when at least three samples exist.
- Otherwise fallback by exercise body part:
  - Large muscle groups: `21` days for `legs`, `back`, `chest`, `full_body`, `full body`.
  - Everything else: `14` days.

Prediction applies rest decay only after a 7 day grace period from the latest workout date:

```ts
decay_factor = exp(-max(0, inactive_days - 7) / personal_tau_days)
```

### Cache

The existing `prediction_models` table stores one cached row per `(user_id, exercise_id)`:

- `slope_per_week`
- `personal_tau_days`
- `sample_size`
- `confidence`
- `fitted_at`

Prediction calls refit and upsert the cache so agents do not need to call a separate setup step. A separate refit job is also exposed for scheduled maintenance.

## Threshold Alarms

### Rule Storage

The existing `alarm_rules` table is used:

- `scope`: `exercise` or `global`
- `exercise_id`: required when `scope = exercise`
- `threshold_type`: `1rm_below`, `days_inactive_above`, `smm_below`, `weight_above`, `weight_below`
- `threshold_value`
- `enabled`
- `last_fired_at`

Validation rules:

- `exercise` rules require `exercise_id`.
- `global` rules must not require an exercise.
- `1rm_below` requires `exercise_id`.
- `days_inactive_above` may be exercise-scoped or global.

### Evaluation

The daily evaluator checks all enabled rules for a user:

- A rule fires at most once per calendar date.
- `1rm_below` compares today's predicted 1RM for the exercise against `threshold_value`.
- `days_inactive_above` compares days since latest workout against `threshold_value`.
- `smm_below` compares latest `body_measurements.smm_kg`.
- `weight_above` and `weight_below` compare latest `body_measurements.weight_kg`.

When a rule fires:

- Write a `notifications_log` row.
- Update `alarm_rules.last_fired_at`.
- Return fired notification details from the evaluator.

### Delivery

Delivery is best-effort and ordered:

1. If `AGENT_WEBHOOK_URL` is configured, POST a JSON alarm payload to it with `AGENT_WEBHOOK_TIMEOUT_MS`.
2. If the agent webhook is missing or fails, and Telegram credentials are configured, call Telegram `sendMessage`.
3. If both delivery paths are unavailable or fail, write `notifications_log.channel = "log_only"` with `generated_by = "fallback"`.

This keeps alarm history intact even when external network delivery is unavailable.

## Implementation Units

- `src/domain/predictions.ts`: fitting, cache upsert, prediction helpers.
- `src/tools/predictions.ts`: prediction tool routes.
- `src/domain/alarms.ts`: rule CRUD, rule evaluation, notification logging.
- `src/lib/notifications.ts`: agent webhook and Telegram delivery.
- `src/tools/alarms.ts`: alarm tool routes.
- `src/jobs/refit-predictions.ts`: one-shot cache refit job.
- `src/jobs/evaluate-alarms.ts`: one-shot alarm evaluator job.

## Acceptance Criteria

- `predict_performance` returns a deterministic estimate and basis for a known workout history.
- `predict_series` returns daily points through `until_date`.
- Prediction calls upsert `prediction_models`.
- Rest decay uses personal tau when at least three eligible rest-return samples exist.
- Rest decay uses fallback tau when samples are insufficient.
- Alarm rules can be created, listed, and disabled through tool routes.
- Alarm evaluator fires each enabled violated rule at most once per day.
- Agent webhook is attempted before Telegram fallback.
- `notifications_log` is written for every fired rule, including log-only fallback.
- Existing tests continue to pass.
